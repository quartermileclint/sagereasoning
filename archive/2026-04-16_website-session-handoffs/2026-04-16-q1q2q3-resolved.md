## Session Purpose
Resolve three open questions carried forward from the Layer 3 wiring session: compiled TS import, context-template factory Layer 3 injection, and evaluate_demo analytics events.

## Decisions Made

### 1. Compiled TS + JSON merge strategy (Q1)
**Decision:** Both files retained. Compiled TS (`project-context-compiled.ts`) is the build-time baseline. JSON (`project-context.json`) overlays on top — JSON wins on overlap.
**Reasoning:** Compiled TS gives type safety, `as const`, and richer fields (positioning, product architecture). JSON gives Supabase dynamic state fallback. The merge order (compiled first, JSON spread on top) means dynamic state always wins, which is correct because Supabase data is fresher.
**Impact:** `project-context.ts` now imports from both sources. The compiled TS baseline includes `positioning` field which enriches the `full` context level.

### 2. Layer 3 injected via factory pattern (Q2)
**Decision:** Layer 3 injection added to `createContextTemplateHandler()` itself (not to each of the 15 individual skill endpoint files).
**Reasoning:** Single injection point means all current and future skills created via the factory automatically receive project context. Reduces per-endpoint maintenance. Default level is `condensed` (~104 tokens) which matches the core API group mapping. Individual skills can override via `projectContextLevel` config field.
**Impact:** All 15+ context-template skills (sage-premortem through sage-moderate) now receive project context automatically. This is a pre-hold-point fix — without it, skill outputs could misrepresent product capabilities.

### 3. Evaluate demo analytics events (Q3)
**Decision:** Three server-side analytics events added inline to `/api/evaluate/route.ts`:
- `evaluate_demo_started` — fires after input validation, before LLM call
- `evaluate_demo_completed` — fires after successful response (or cache hit), includes `from_cache` flag
- `evaluate_demo_error` — fires in catch block, includes error type class name only
**Reasoning:** Hold point assessment needs demo flow visibility for developer acquisition metrics. Events are fire-and-forget (never block demo response), use hashed IP (privacy-preserving), and contain no PII. The `from_cache` flag distinguishes repeat evaluations from fresh ones.
**Impact:** P0 hold point blocker resolved. Demo funnel now has start/complete/error visibility.

### 4. Analytics table schema mismatch fix
**Decision:** Removed `ip_address` and `user_agent` from all analytics inserts in the evaluate route.
**Reasoning:** The `analytics_events` table only has columns: `id`, `event_type`, `user_id`, `metadata`, `created_at`, `api_key_id`. The initial insert code included `ip_address` and `user_agent` fields (modelled on `/api/analytics/route.ts`), but those columns don't exist on the table. Supabase rejected the inserts silently because the fire-and-forget pattern swallowed errors.
**Impact:** Analytics events now land correctly. Verified in production.

### 5. Evaluate demo resilience fix
**Decision:** Made `katorthoma_proximity`, `philosophical_reflection`, and `improvement_path` non-fatal — they get sensible defaults if the LLM omits them. The three core mechanism fields (`control_filter`, `passion_diagnosis`, `oikeiosis`) remain hard requirements.
**Reasoning:** The LLM occasionally omits summary fields, which caused the demo to return a 500 error. A crashing demo is worse than a partial-but-useful response.
**Impact:** The public demo endpoint no longer 500s on missing summary fields.
